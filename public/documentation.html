1 System Overview

The Agricultural Services Database is a relational database system built using MySQL within VS Code to support Uganda's Ministry of Agriculture in managing coffee production, farmer registration, input distribution, and export traceability. The system addresses the full coffee supply chain — from registering a farmer and their garden through to tracking exports.

Core Objective: Enable the Ministry to know every coffee grower, track their production season by season, distribute agricultural inputs, coordinate extension worker visits, and generate ministry-level production and export reports.

Key Functional Capabilities

CapabilityDescription
Farmer RegistrationRecord all coffee growers with their NIN-linked personal details, farm, and garden information.
Production TrackingRecord harvests per garden per season including grade, quantity (kg), and production costs.
Input DistributionPlan and record delivery of seedlings, fertilizers, pesticides, and tools to verified gardens.
Extension ServicesLog farm visits, raise resource requests, and link workers to specific gardens.
Processing & ExportTrack raw-to-processed coffee conversion and export batches with USD values.
Location Hierarchy4-level geographic model: Region → District → Subcounty → Village.
ReportingMinistry-level views for regional production, export totals, and yield analysis.

2 Database Architecture

The database is organized into seven thematic layers, each grouping related entities. This layering mirrors the real-world domains of the Ministry system:

LayerColorKey EntitiesPurpose
People Purple Person, Farmer, Staff, ExtWorker, DBManager, SystemUser All human actors — superclass/subtype hierarchy
Farm / Garden Green Farm, Garden, Cooperative, CoopMember, GardenSeason Physical assets — farms, plots, cooperatives
Production Orange Season, ProductionRecord, SaleRecord, CoffeeType, CoffeeGrade, MarketPrice Harvest recording and price management
Distribution Red Distribution, DistributionItem, Inputs, InputInventory, ReceivedInputs, InputCategory Agricultural input stock and delivery
Export Blue Processor, ProcessingRecord, Exporter, ExportBatch Post-harvest processing and international export
Admin Grey License, LicenseType, Region, District, Subcounty, Village, Office Licensing, location, and office management
Support Pink FarmVisit, ResourceRequest, Scheme, SchemeEnrolment, CropMarketPrice Extension visits, government schemes, and market data

Supply Chain Data Flow

End-to-end traceability:
Village → Person → Farmer → Farm → Garden → ProductionRecord → ProcessingRecord → ExportBatch
Every kilogram of coffee exported can be traced back to the specific garden and farmer who grew it.

4 EER Design Features

Specialization / Generalization

HierarchyTypeRule
Person → {Farmer, Staff, SystemUser} OVERLAPPING (o) A Person can be both a SystemUser AND a Farmer or Staff. SystemUser overlaps with all subtypes.
Staff → {ExtensionWorker, DatabaseManager} DISJOINT (d) A Staff member is EITHER a field Extension Worker OR a Database Manager — never both.
Inputs → {InputInventory, ReceivedInputs} DISJOINT (d) An Input record is EITHER in store inventory OR a new shipment — not both simultaneously.
Processor / Exporter (ServiceProvider) OVERLAPPING (o) A company can be both a Processor AND an Exporter (e.g. vertically integrated factory).

Attribute Inheritance via Triggers

Farm.Acreage is never entered directly — it is computed as SUM(Garden.Acreage) for all gardens belonging to that farm. The trigger AfterGardenInsertUpdate maintains this automatically.

ProcessingRecord.LossKg and YieldPercent are generated columns maintained by the AfterProcessingRecord trigger, ensuring they are always consistent with RawKgIn and ProcessedKgOut.

5 Triggers

Triggers implement automated business rules that fire in response to data change events. The system uses 12 triggers across BEFORE and AFTER INSERT/UPDATE events.

AfterGardenInsertUpdate AFTER INSERT / UPDATE ON Garden
Purpose: Maintains Farm.Acreage as the sum of all garden acreages for that farm. Also sets Farm.IsGroup = TRUE if more than one cooperative member is linked to the farm. 
DELIMITER //
CREATE TRIGGER AfterGardenInsertUpdate
AFTER INSERT ON Garden
FOR EACH ROW
BEGIN
  UPDATE Farm
  SET Acreage = (
    SELECT SUM(Acreage) FROM Garden WHERE FarmID = NEW.FarmID
  )
  WHERE FarmID = NEW.FarmID;
END//
DELIMITER ;
BeforeDistributionItemInsert BEFORE INSERT ON DistributionItem
Purpose: Validates that Inputs.Qty >= QuantityGiven before any item is distributed. Raises SQLSTATE 45000 to abort the insert if stock is insufficient. Prevents over-distribution before it occurs. 
CREATE TRIGGER BeforeDistributionItemInsert
BEFORE INSERT ON DistributionItem
FOR EACH ROW
BEGIN
  DECLARE available DECIMAL(10,2);
  SELECT Qty INTO available FROM Inputs WHERE InputID = NEW.InputID;
  IF available < NEW.QuantityGiven THEN
    SIGNAL SQLSTATE '45000'
    SET MESSAGE_TEXT = 'Insufficient stock for this input item.';
  END IF;
END//
AfterDistributionItemInsert AFTER INSERT ON DistributionItem
Purpose: Deducts QuantityGiven from Inputs.Qty after a successful distribution item insert. Also inserts a record into DistributionLog for full audit trail. 
CREATE TRIGGER AfterDistributionItemInsert
AFTER INSERT ON DistributionItem
FOR EACH ROW
BEGIN
  UPDATE Inputs
  SET Qty = Qty - NEW.QuantityGiven
  WHERE InputID = NEW.InputID;

  INSERT INTO DistributionLog (DistribID, InputID, QuantityGiven, LogTime)
  VALUES (NEW.DistribID, NEW.InputID, NEW.QuantityGiven, NOW());
END//
ValidatePlotOwnership BEFORE INSERT ON Distribution
Purpose: Verifies that Distribution.GardenID → Garden.FarmID → Farmer.FarmerID matches Distribution.FarmerID. Prevents recording a distribution against a garden that does not belong to the specified farmer.
AfterDistributionStatusChange AFTER UPDATE ON Distribution
Purpose: When DeliveryStatus changes to 'Dispatched', automatically sets DispatchTime = NOW(). When it changes to 'Delivered', logs the event to DeliveryLog. Enforces automatic timestamp management. 
CREATE TRIGGER AfterDistributionStatusChange
AFTER UPDATE ON Distribution
FOR EACH ROW
BEGIN
  IF NEW.DeliveryStatus = 'Dispatched' AND OLD.DeliveryStatus != 'Dispatched' THEN
    UPDATE Distribution SET DispatchTime = NOW()
    WHERE DistribID = NEW.DistribID;
  END IF;
END//
BlockUnverifiedGardenHarvest BEFORE INSERT ON ProductionRecord
Purpose: Checks Garden.IsVerified = TRUE before allowing a harvest record. If the garden has not yet been verified by an extension worker, the insert is blocked. Enforces the farm verification business rule. 
CREATE TRIGGER BlockUnverifiedGardenHarvest
BEFORE INSERT ON ProductionRecord
FOR EACH ROW
BEGIN
  DECLARE verified BOOLEAN;
  SELECT IsVerified INTO verified FROM Garden WHERE GardenID = NEW.GardenID;
  IF verified = FALSE THEN
    SIGNAL SQLSTATE '45000'
    SET MESSAGE_TEXT = 'Garden must be verified before recording a harvest.';
  END IF;
END//
GardenSeasonUniqueness BEFORE INSERT ON GardenSeason
Purpose: Enforces the rule that one garden can only be in one active season at a time. Raises an error if a GardenID already has an active SeasonID record in GardenSeason. 
CREATE TRIGGER GardenSeasonUniqueness
BEFORE INSERT ON GardenSeason
FOR EACH ROW
BEGIN
  IF EXISTS (
    SELECT 1 FROM GardenSeason
    WHERE GardenID = NEW.GardenID AND SeasonID = NEW.SeasonID
  ) THEN
    SIGNAL SQLSTATE '45000'
    SET MESSAGE_TEXT = 'Garden already registered for this season.';
  END IF;
END//
AfterProcessingRecord AFTER INSERT ON ProcessingRecord
Purpose: Calculates LossKg = RawKgIn - ProcessedKgOut and YieldPercent = (ProcessedKgOut / RawKgIn) × 100. Updates these on the newly inserted row and logs to ProcessingAuditLog. 
CREATE TRIGGER AfterProcessingRecord
AFTER INSERT ON ProcessingRecord
FOR EACH ROW
BEGIN
  UPDATE ProcessingRecord
  SET
    LossKg        = NEW.RawKgIn - NEW.ProcessedKgOut,
    YieldPercent  = (NEW.ProcessedKgOut / NEW.RawKgIn) * 100
  WHERE ProcessID = NEW.ProcessID;
END//
AfterExportBatchInsert AFTER INSERT ON ExportBatch
Purpose: Calculates TotalUSD = QuantityKg × USDPricePerKg and updates the row. Also updates a SeasonExportSummary table with cumulative export totals by season and grade for ministry reporting.
LicenseExpiryCheck BEFORE INSERT ON Distribution / ExportBatch
Purpose: Verifies that the related License.ExpiryDate > CURRENT_DATE and License.Status = 'Active'. Blocks distribution or export operations if the license is expired or suspended.
ValidateStaffRegion BEFORE INSERT ON ExtensionWorker
Purpose: Prevents a staff member from being assigned to their own home region. Traverses Person → Village → Subcounty → District → Region to find the staff member's home region and compares it with the assigned RegionID.
FarmCoffeeVarietyUpdate AFTER INSERT / UPDATE ON Garden
Purpose: Updates Farm.CoffeeVariety with a concatenated, distinct list of coffee grade names across all gardens belonging to that farm. Keeps the farm-level variety summary always current.

6 Stored Procedures

Stored procedures encapsulate complex multi-step operations, enforce business rules in code, and promote reuse. The system includes 12 stored procedures.

RegisterFarmer(IN p_NIN, p_FirstName, p_LastName, p_DOB, p_Phone1, p_VillageID, p_FarmID)
Inserts into Person then Farmer in a single transaction. Returns the new FarmerID. Validates that no existing Farmer or Staff record exists for the NIN (disjoint constraint enforcement). 
DELIMITER //
CREATE PROCEDURE RegisterFarmer(
  IN p_NIN VARCHAR(20), IN p_FirstName VARCHAR(50),
  IN p_LastName VARCHAR(50),  IN p_VillageID INT,
  IN p_FarmID INT
)
BEGIN
  START TRANSACTION;
  INSERT INTO Person (NIN, FirstName, LastName, VillageID)
  VALUES (p_NIN, p_FirstName, p_LastName, p_VillageID);

  INSERT INTO Farmer (NIN, FarmID) VALUES (p_NIN, p_FarmID);
  COMMIT;
  SELECT LAST_INSERT_ID() AS NewFarmerID;
END//
DELIMITER ;
RecordHarvest(IN p_GardenID, p_SeasonID, p_DateHarvested, p_HarvestedKg, p_ProductionCost)
Checks that GardenSeason exists (garden enrolled in that season) and Garden.IsVerified = TRUE before inserting a ProductionRecord. Returns ProdID.
DistributeToGarden(IN p_FarmerID, p_GardenID, p_WorkerID, p_Items JSON)
Creates a Distribution plus multiple DistributionItem records in one call. Accepts a JSON array of {InputID, Quantity} pairs. Validates garden ownership and stock levels for all items before any insert (all-or-nothing). Trigger then deducts stock.
ApproveResourceRequest(IN p_RequestID, p_ApproverStaffID)
Updates ResourceRequest.Status to 'Approved', then internally calls DistributeToGarden to fulfil the request. Both operations are wrapped in one transaction — if distribution fails, the approval is rolled back.
RecordProcessing(IN p_GardenID, p_ProcessorID, p_SeasonID, p_RawKgIn, p_ProcessedKgOut, p_ProcessType)
Inserts a ProcessingRecord. Validates that the ProcessorID holds a valid, non-expired License. Trigger then calculates LossKg and YieldPercent automatically.
CreateExportBatch(IN p_ExporterID, p_ProcessID, p_SeasonID, p_QuantityKg, p_DestCountry, p_USDPricePerKg)
Inserts an ExportBatch. Validates that ExporterID has an active UCDA license and that QuantityKg ≤ ProcessingRecord.ProcessedKgOut (remaining). Trigger calculates TotalUSD.
VerifyGarden(IN p_GardenID, p_WorkerID)
Sets Garden.IsVerified = TRUE, records the verifying WorkerID, and creates a FarmVisit record for the verification event. Wrapped in a transaction.
EnrolFarmerInScheme(IN p_FarmerID, p_SchemeID)
Inserts into SchemeEnrolment. Checks the scheme is still active (EndDate > NOW()) and that the farmer is not already enrolled. Optionally triggers an initial distribution if the scheme provides inputs on enrolment.
GetFarmerProductionReport(IN p_FarmerID) → ResultSet
Returns all ProductionRecords, SaleRecords, and ProcessingRecords for a given farmer across all seasons and gardens. Includes totals and grade breakdown for ministry reporting.
GetSeasonExportSummary(IN p_SeasonID) → ResultSet
Returns total KG harvested, processed, sold locally, and exported per season. Includes yield loss percentage and average export price. Primary tool for ministry-level reporting.
TransferGardenOwnership(IN p_GardenID, p_NewFarmID)
Updates Garden.FarmID, inserts a PlotHistory record, and recalculates acreage on both the old and new Farm. Wrapped in a transaction.
RegisterGarden(IN p_FarmID, p_GPS, p_VillageID, p_Acreage, p_GradeID, p_PlantingDate, p_NumberOfTrees)
Creates a Garden record under a Farm. Validates that FarmID belongs to the calling FarmerID. The AfterGardenInsertUpdate trigger automatically recalculates Farm.Acreage.

7 Transaction Management

All multi-step data operations are wrapped in explicit BEGIN / COMMIT / ROLLBACK transactions to guarantee atomicity. If any step fails, the entire operation is reversed.

TransactionStepsEnsures
New Farmer Registration INSERT Person → INSERT Farmer → INSERT Farm → INSERT Garden → INSERT GardenSeason → COMMIT A new farmer must have at least one farm and garden. No orphaned Person records.
Harvest to Sale INSERT ProductionRecord → INSERT ProcessingRecord → INSERT SaleRecord → COMMIT A harvest is never partially recorded across the supply chain.
Approve & Fulfil Resource Request UPDATE ResourceRequest (Approved) → INSERT Distribution → INSERT DistributionItem(s) → deduct stock → UPDATE ResourceRequest (Fulfilled) → COMMIT Approval and distribution are atomic — approval never exists without fulfilment or vice versa.
Export Batch Creation Validate ProcessingRecord stock → INSERT ExportBatch → UPDATE remaining stock → INSERT ExportAuditLog → COMMIT Cannot export more than was processed.
Cooperative Formation INSERT Cooperative → INSERT CooperativeMember for each farmer → UPDATE Farm linkages → COMMIT Entire cooperative creation is atomic.
Season Closure & Rollover UPDATE Season.EndDate → verify all GardenSeason records closed → INSERT new Season → COMMIT Prevents two open seasons simultaneously across the system.
Bulk Scheme Distribution For each enrolled farmer → call DistributeToGarden → if any fails stock check → ROLLBACK ALL → else COMMIT Ministry-wide input distribution is all-or-nothing across all enrolled farmers.
Garden Split (Land Heritage) UPDATE old Garden status → INSERT two new Gardens → INSERT PlotHistory → trigger recalculates Farm.Acreage → COMMIT Land subdivision is fully tracked historically.

8 Views

Views provide pre-built, read-only query abstractions for ministry reporting without exposing raw table structure.

Ministry_Regional_Production

Shows total KG harvested and distinct active farmers per District per Season. Used by ministry administrators for regional production oversight.

CREATE VIEW Ministry_Regional_Production AS
SELECT
  d.Name            AS District,
  s.Name            AS Season,
  SUM(pr.HarvestedKg) AS Total_Kgs,
  COUNT(DISTINCT f.FarmerID) AS Active_Farmers
FROM ProductionRecord pr
JOIN Season s    ON pr.SeasonID  = s.SeasonID
JOIN Garden g    ON pr.GardenID  = g.GardenID
JOIN Farm farm   ON g.FarmID     = farm.FarmID
JOIN Farmer f    ON farm.FarmID  = f.FarmID
JOIN Person p    ON f.NIN        = p.NIN
JOIN Village v   ON p.VillageID  = v.VillageID
JOIN Subcounty sc ON v.SubcountyID = sc.SubcountyID
JOIN District d  ON sc.DistrictID = d.DistrictID
GROUP BY d.Name, s.Name;
ViewPurpose
Ministry_Regional_ProductionRegional harvest totals and farmer counts per season
FarmerSupplyChainEnd-to-end trace for each farmer: garden → harvest → processing → export
ActiveDistributionsAll distributions with DeliveryStatus ≠ 'Delivered'
LicenseExpiryAlertAll licenses expiring within 30 days
SeasonExportSummaryCumulative export KG, USD value, and average price per grade per season
ExtWorkerPerformanceVisit count, gardens verified, and requests raised per extension worker

9 Security & Access Control

Security is implemented through MySQL's native user account system combined with role-based privilege grants. Passwords are stored as bcrypt hashes in the SystemUser table.

User Roles & Privileges

RoleMySQL UserGranted PrivilegesRestricted Tables
Ministry Admin 'admin'@'localhost' ALL PRIVILEGES on agri_db.* None — full access
Database Manager 'db_manager'@'localhost' SELECT, INSERT, UPDATE, DELETE on all tables; CREATE/DROP procedures and views Cannot DROP tables
Extension Worker 'ext_worker'@'localhost' SELECT on all tables; INSERT on FarmVisit, ResourceRequest; UPDATE on Garden (IsVerified only) No access to License, ExportBatch, SystemUser, Distribution financials
Farmer (Self-Service) 'farmer_user'@'%' SELECT on own ProductionRecord, SaleRecord, Distribution (via views filtered by FarmerID) Cannot see other farmers' data; no access to Staff, SystemUser, License
Read-Only Reporter 'reporter'@'localhost' SELECT on views only (Ministry_Regional_Production, SeasonExportSummary) No base table access 
-- Create extension worker user
CREATE USER 'ext_worker'@'localhost'
  IDENTIFIED BY 'StrongPassword#2026';

GRANT SELECT ON agri_db.* TO 'ext_worker'@'localhost';
GRANT INSERT ON agri_db.FarmVisit TO 'ext_worker'@'localhost';
GRANT INSERT ON agri_db.ResourceRequest TO 'ext_worker'@'localhost';
GRANT UPDATE (IsVerified) ON agri_db.Garden TO 'ext_worker'@'localhost';
FLUSH PRIVILEGES;

Additional Security Measures

Authentication: SystemUser.PasswordHash stores bcrypt-hashed passwords. Raw passwords are never stored or logged in the database. Application layer handles authentication before issuing SQL connections.
Views as Security Layer: Farmer-facing queries use views filtered by FarmerID or NIN, ensuring each farmer sees only their own data. The base tables are not directly accessible to farmer user accounts.

10 Backup & Recovery Strategy

Backup Methods

MethodToolFrequencyOutput
Full Logical Dump mysqldump Daily (end of day) agri_db_YYYYMMDD.sql — complete DDL + data
Structure-Only Dump mysqldump --no-data After any schema change agri_db_schema.sql — DDL only for version control
Incremental Backup MySQL Binary Log Continuous (binlog enabled) Binary log files for point-in-time recovery
Export to CSV SELECT INTO OUTFILE Weekly — production data only Human-readable data for offline archival

Backup Commands

-- Full logical backup (run from terminal)
mysqldump -u admin -p agri_db > agri_db_backup_$(date +%Y%m%d).sql

-- Structure only (for schema version control)
mysqldump -u admin -p --no-data agri_db > agri_db_schema.sql

-- Restore from dump file
mysql -u admin -p agri_db < agri_db_backup_20260412.sql

Recovery Procedure

ScenarioRecovery ActionExpected RTO
Accidental row deletionRestore from same-day dump; replay binary log to point-in-time< 30 minutes
Full database corruptionRestore latest full dump; replay binary log since last dump< 2 hours
Schema change errorRoll back using structure-only backup; re-apply data migr.< 1 hour
Server failureRestore dump to new MySQL instance; update connection strings< 4 hours
Best Practice: All backup files are stored in a separate, off-site location (e.g. USB drive or cloud storage). Backup integrity is verified monthly by running a test restore to a development instance and executing CHECK TABLE on all tables.
