Business Rules

Governance Rules & EERD Assumptions

Every constraint in the database reflects a real regulatory or operational requirement from Uganda's coffee sector.

Rule Description Enforcement
Mandatory Registration Every farmer must have a valid NIN and be registered with at least one farm before any garden can record production.[BR-01] Schema FK + Procedure RegisterFarmer
Certified Seedlings Only All distributed seedlings must come from UCDA-certified sources; stock levels are validated before any distribution.[BR-02] Trigger BeforeDistributionItemInsert
Immature Cherry Prohibition Gardens must be verified by an extension worker before any harvest can be recorded — preventing premature cherry collection.[BR-03] Trigger BlockUnverifiedGardenHarvest
Single Coffee Type per Garden Each garden is assigned exactly one CoffeeGrade (linked to one CoffeeType), enforcing the rule that a garden grows only Arabica or Robusta.[BR-04] Schema FK Garden.GradeID
Regional Worker Restriction Extension workers are assigned to a specific region and can only perform visits and distributions within that region.[BR-05] Schema FK ExtWorker.RegionID
Export Volume Guard Cannot export more kilograms than were processed — cumulative export batches validated against ProcessingRecord.[BR-06] Procedure CreateExportBatch

EERD Concepts

Specialization &
Generalization

Specialization decomposes a supertype entity into meaningful subtypes, each inheriting the supertype's attributes while adding their own specialized fields.[Elmasri & Navathe]

Supertype Subtypes Constraint
Person Farmer, Staff, SystemUser Farmer/Staff: Disjoint
SystemUser: Overlapping
Staff ExtWorker, DBManager Disjoint
Inputs InputInventory, ReceivedInputs Disjoint
Attribute Inheritance
A Farmer inherits NIN, FirstName, LastName, Phone1, and VillageID from Person.

The Farmer subtype adds FarmerID (auto-generated PK) and a mandatory FarmID foreign key — binding every farmer to a registered farm.[00_schema.sql]
Disjoint vs Overlapping
Disjoint: A Person cannot be both a Farmer and a Staff member simultaneously — enforced by separate FK constraints on the unique NIN.

Overlapping: Any Person subtype can have SystemUser credentials — a Farmer can also log in to the system.[00_schema.sql]

EERD Concepts

Categorization &
Union Types

A categorization (union type) models a relationship where a single entity's supertype can come from one of multiple independent entity types. In our system, a License holder can be either a Person or a Cooperative.[Elmasri & Navathe]

-- License uses CATEGORIZATION UNION CREATE TABLE License ( LicenseID INT PRIMARY KEY, HolderNIN VARCHAR(20), -- Person CoopRegNo VARCHAR(20), -- Cooperative HolderType ENUM('Person','Cooperative'), Status ENUM('Active','Expired',...) );
Overlapping: Processor & Exporter
A company can be both a coffee Processor and an Exporter simultaneously. Both entities share a LicenseID foreign key but operate independently — the same company can process and export its own coffee.[00_schema.sql]
Bridge Entities (M:N Resolution)
CooperativeMember: Resolves M:N between Cooperative and Farmer
GardenSeason: Links gardens to harvest seasons
DistributionItem: One distribution event carries many input types
SchemeEnrolment: M:N between Farmer and government Scheme[00_schema.sql]

Automation — Stored Procedures

Core Procedures[02_procedures.sql]

RegisterFarmer
Atomic transaction: INSERT Person → INSERT Farmer. Wrapped in EXIT HANDLER for automatic ROLLBACK on failure. Returns new FarmerID via OUT parameter.[P1]
DistributeToGarden
JSON-driven bulk distribution. Pre-validates ALL stock levels before touching any data, then inserts Distribution + DistributionItems. Triggers auto-deduct stock.[P5]
CreateExportBatch
Validates exporter license status, checks cumulative export volume against processed stock. Trigger auto-updates SeasonExportSummary.[P8]
-- P5: DistributeToGarden (JSON-driven) CREATE PROCEDURE DistributeToGarden( IN p_FarmerID INT, IN p_GardenID INT, IN p_WorkerID INT, IN p_Items JSON, OUT p_DistribID INT ) BEGIN -- Pre-validate ALL items WHILE v_i < v_len DO SELECT Qty INTO v_available FROM Inputs WHERE InputID = v_InputID; IF v_available < v_Qty THEN SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'Insufficient stock.'; END IF; END WHILE; START TRANSACTION; INSERT INTO Distribution ...; -- Loop INSERT DistributionItem COMMIT; END;

Backup & Recovery

Data Protection Strategy

A multi-layered backup strategy ensures data durability and quick recovery from any failure scenario.

mysqldump — Logical Backup
Weekly full dump of all 38 tables including triggers and procedures. Compressed with gzip. Retention: 4 weekly snapshots rotated.[Backup Policy]
Binary Log Replication
Point-in-time recovery enabled via MySQL binary logs. Combined with weekly full backups for sub-hour RPO.[MySQL Docs]
Weekly Redundancy Schedule
Automated cron job: Sunday 02:00 AM EAT. Backup file transmitted to off-site cold storage. Alert on failure via email notification.[Backup Policy]
# Weekly full backup script mysqldump \ --routines --triggers --events \ --single-transaction \ --quick \ agri_db | gzip > \ /backup/agri_db_$(date +%F).sql.gz # Verify backup integrity gunzip -t /backup/agri_db_*.sql.gz # Restore from backup gunzip < backup.sql.gz | \ mysql agri_db

Conclusion

Scalable, Secure, and
Standards-Compliant

The Agricultural Services Database demonstrates a complete, production-ready data platform that could scale from district-level pilot to national deployment.

Scalability
Modular SQL architecture (schema → triggers → procedures → views → users) allows independent scaling of each layer. InnoDB engine provides row-level locking for concurrent access.[Architecture]
Security
Five-tier RBAC with column-level GRANT privileges. Stored procedures encapsulate business logic — users execute procedures, not raw SQL. Audit trails via trigger-driven logs.[05_users.sql]
Data Integrity
12 triggers enforce business rules at the database level — impossible to bypass via application code. ACID transactions guarantee all-or-nothing operations.[01_triggers.sql]
Deliverables Summary
38 tables, 12 triggers, 12 procedures, 6 views, 5 user roles, 8 transaction scripts, interactive EERD visualizer, SQL demo guide, technical documentation, and this presentation.
